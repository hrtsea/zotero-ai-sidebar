把“原文”从一个按钮，拆成清晰的上下文模式

适配目标：普通 PDF 仍然显示“原文”；arXiv 源论文不再用“原文”误导用户，而显示“目录模式”。 需要完整正文时提供显式的“本轮全文”入口，并在调试区直接说明本轮实际发送了目录、全文，还是由模型调用工具读取。

效果总结

解决歧义：arXiv 模式下“原文”不再暗示已发送完整正文。
保留省 token：默认仍发送章节目录 + 工具，适合多数问答。
可强制确定：“本轮全文”用于总结全文、审稿、全面对比等任务。
调试可验证：状态条展示 Front Block 与工具调用，避免把历史回答误当成本轮原文。

FLOW B - 用户看到的因果关系

从“我以为发了原文”变成“我知道发了什么”

当前容易误解

按钮：📄 原文 调试：已随本轮发送 arXiv 章节目录 2074 字 模型：可能调用全文，也可能不调用 用户感受： 为什么点了“原文”，第一章内容却没有发送？ 为什么有原文引用，但 Front Block 只有 2074 字？

适配后更清楚

按钮：📑 目录 ＋ 本轮全文 调试：已随本轮发送 arXiv 章节目录 2074 字 工具：arxiv_get_section / zotero_get_full_pdf 用户判断： 目录模式 = 没有完整正文； 需要全文总结时点“本轮全文”。

注意：这只是 UI 适配效果稿，暂不改变后端路由。真正实现时可以先做最小改动：只改按钮文案、tooltip、状态提示； 后续再加确定性路由，例如“总结全文”自动强制 full text，“第 1 章”自动读取 section 1。

推荐落地范围

先做低风险 UI 语义修正，再考虑自动路由

第一阶段：只改展示，不改模型策略

  • PDF 源：保持“＋ 原文 / 📄 原文”。
  • arXiv 源：改成“＋ 目录 / 📑 目录”。
  • tooltip 明确：目录模式发送章节索引，正文由工具按需读取。
  • 增加“本轮全文”入口，但默认不长期固定全文，避免 cache 与 token 成本恶化。

第二阶段：再加确定性任务路由

  • “总结全文 / 全文审稿 / 全面比较”：强制调用 full text。
  • “第一章 / Introduction / 5.3”：强制读取对应 arXiv section。
  • 调试记录里写明是本地强制，还是模型自主调用。
  • 保留模型按需调用能力，但不要让它承担所有上下文选择责任。
